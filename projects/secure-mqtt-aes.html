Transmission sécurisée de données via MQTT

Transmission de données environnementales simulées entre un équipement et une entreprise, via un broker MQTT, avec un chiffrement bout-en-bout. Le broker relaie les messages sans jamais pouvoir les lire en clair.

CatégorieCybersécurité / Dev
TypeSAE (projet académique)
ÉquipeBinôme
Langage principalJava 17

Contexte

Ce projet a été réalisé dans le cadre d'une SAE (Situation d'Apprentissage et d'Évaluation) du cursus ingénieur à l'ESIROI. L'objectif était de simuler la transmission de données issues d'un équipement connecté (température, humidité…) vers un poste « entreprise », en passant par un broker MQTT, tout en garantissant la confidentialité des données — y compris vis-à-vis du broker.

Objectifs

  • Comprendre le fonctionnement d'un système de messagerie MQTT (publish / subscribe).
  • Implémenter un chiffrement bout-en-bout des messages avec AES.
  • Appliquer une dérivation de clé robuste à partir d'un secret partagé (PBKDF2).
  • Structurer une application Java modulaire et isoler la configuration du code métier.

Technologies utilisées

  • Java 17
  • Eclipse Paho MQTT
  • org.json
  • Mosquitto
  • AES-CBC
  • PBKDF2-HMAC-SHA256
  • SHA-256
  • Ubuntu Server / Desktop

Travaux réalisés

  • Mise en place d'un broker MQTT local (Mosquitto) et validation de la chaîne publish/subscribe entre les machines.
  • Développement de trois classes Java : CryptoUtils (dérivation de clé et chiffrement), ClientPublisher (équipement émetteur) et ClientSubscriber (récepteur).
  • Chiffrement des messages JSON en AES-CBC avant publication, déchiffrement côté récepteur.
  • Dérivation du topic MQTT à partir d'un identifiant partagé (room) via SHA-256, pour cloisonner les échanges.
  • Externalisation de l'adresse du broker dans un fichier de configuration, pour éviter tout codage en dur et faciliter le déploiement multi-machine.

Architecture & fonctionnement

Le système repose sur trois machines virtuelles Ubuntu : un émetteur (équipement), un récepteur (entreprise) et un broker Mosquitto, sur un même réseau local. La clé AES est dérivée du secret room par PBKDF2-HMAC-SHA256 ; seuls les clients partageant ce même secret peuvent reconstituer la clé et le topic. L'émetteur chiffre un message JSON puis le publie ; le récepteur, abonné au topic dérivé, déchiffre et affiche le contenu. Le broker ne manipule que des données chiffrées.

Résultats obtenus

La chaîne complète est fonctionnelle : un message chiffré est publié par l'équipement, relayé par le broker sans être lisible, puis correctement déchiffré et affiché côté entreprise. L'objectif de confidentialité bout-en-bout est atteint dans le périmètre du projet.

Difficultés rencontrées

  • Mettre en place une dérivation de clé correcte et cohérente entre l'émetteur et le récepteur.
  • Construire un topic sécurisé dérivé du secret partagé.
  • Séparer proprement les responsabilités (chiffrement, réseau, configuration) pour garder un code maintenable.

Compétences développées

  • Cryptographie appliquée : chiffrement symétrique AES, dérivation de clé PBKDF2, hachage SHA-256.
  • Protocoles de messagerie : MQTT et modèle publish/subscribe.
  • Développement Java structuré et gestion de la configuration.
  • Administration de machines Linux et mise en réseau local.

Améliorations possibles

  • Génération d'un sel aléatoire par message.
  • Vérification d'intégrité des messages (HMAC).
  • Stockage des données reçues côté entreprise (base de données).
  • Ajout d'une authentification des clients.
